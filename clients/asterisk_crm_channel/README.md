# asterisk_crm_channel

Asterisk CRM bridge for call events.

## Integration key

- `asterisk_crm_channel`

## Required settings (ConnectedIntegrationSetting)

- `asterisk_enabled` (`bool`)
- `asterisk_account_key` (`string`)
- `asterisk_base_url` (`string`)
- `asterisk_ari_app` (`string`)
- `asterisk_ari_user` (`string`)
- `asterisk_ari_password` (`string`)
- `asterisk_pipeline_id` (`int > 0`)
- `asterisk_channel_id` (`int > 0`)

## Optional settings

- `asterisk_default_responsible_user_id` (`int > 0`)
- `asterisk_lead_subject_template` (`string`, default: `Call {direction} {from_phone}`)
- `asterisk_allowed_did_list` (`string`, CSV)
- `asterisk_recording_base_url` (`string`)
- `lead_dedupe_ttl_sec` (`int >= 60`, default: `86400`)
- `state_ttl_sec` (`int >= 60`, default: `86400`)
- `reconcile_lookback_min` (`int >= 1`, default: `120`)

## External endpoint

- `POST /clients/asterisk_crm_channel/external`
- header: `connected-integration-id: <id>`

Body can be:

- single event object
- `{ "event": {...} }`
- `{ "events": [{...}, {...}] }`
- array of events

Minimal normalized fields after parsing:

- `external_call_id`
- `status` (`started|ringing|answered|missed|completed|failed|recording_ready`)
- `direction` (`inbound|outbound`)
- `from_phone`
- `to_phone`
- `client_phone`
- `event_ts`
- `event_id` (if absent, generated by hash)

## Redis keys

Prefix: `clients:asterisk_crm_channel:`

- `settings:{connected_integration_id}`
- `stream:asterisk_in:{connected_integration_id}`
- `stream:dlq:{connected_integration_id}`
- `worker:heartbeat:{connected_integration_id}:{instance_id}`
- `lock:create_lead:{connected_integration_id}:{asterisk_hash}:{normalized_phone}`
- `dedupe:event:{connected_integration_id}:{event_id}`
- `mapping:by_phone:{connected_integration_id}:{asterisk_hash}:{normalized_phone}`
- `mapping:by_call:{connected_integration_id}:{asterisk_hash}:{external_call_id}`
- `late_recording:{connected_integration_id}:{asterisk_hash}:{external_call_id}`

TTL policy:

- runtime keys always use explicit TTL
- no `TTL=1`
- no persistent runtime keys (`TTL=-1`)

## Lead logic

- lead lookup uses active statuses only: `New/InProgress/WaitingClient`
- `external_contact_id`: `ast:{asterisk_hash}:{normalized_phone}`
- if active lead is absent, `Lead/Add` is called (protected by distributed lock)

## 1220 handling

`ChatMessage/Add` returns `error=1220` (closed linked entity):

- call-state statuses: clear mapping -> resolve/create active lead -> one retry
- `recording_ready`: mark event as `late_recording_closed_lead`, store recording metadata, do not create/reopen lead

## Lead status policy

- `InProgress`:
  - inbound: `started|ringing|answered`
  - outbound: `answered`
- `WaitingClient`:
  - outbound: `started|ringing`

Terminal statuses (`Closed`, `Converted`) are not overwritten.
